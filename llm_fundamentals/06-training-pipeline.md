# 06 — Training Pipeline

Goal: understand, at an engineer's depth, how a modern LLM is trained. The three-stage pipeline (pretraining → SFT → RLHF/RLAIF), the loss functions, what each stage actually teaches, and why the field has moved toward DPO, constitutional AI, and distillation.

You will not be able to fine-tune a frontier model after reading this. You *will* be able to reason about why a model behaves the way it does — which stage introduced which capability.

---

## 1. The three stages, in one diagram

```
  Internet-scale raw text
         │
         ▼
  ┌──────────────────────┐
  │    PRETRAINING       │   Next-token prediction on trillions of tokens.
  │                      │   Produces a "base model" — good at completion,
  │                      │   bad at following instructions.
  └──────────┬───────────┘
             │
             ▼
  ┌──────────────────────┐
  │  SUPERVISED FINE-    │   Train on curated (instruction, response) pairs.
  │  TUNING (SFT)        │   Teaches the model the *format* of answering.
  └──────────┬───────────┘
             │
             ▼
  ┌──────────────────────┐
  │  PREFERENCE TUNING   │   RLHF, DPO, or constitutional AI.
  │  (RLHF / DPO / RLAIF)│   Aligns the model toward human-preferred answers
  │                      │   beyond what supervised data alone can express.
  └──────────┬───────────┘
             │
             ▼
     Production chat model
```

The base model and the final chat model are the **same neural network with different weights**. Each stage is a continuation of training — fewer examples, more careful, with a different objective.

## 2. Stage 1: Pretraining

### 2.1 The objective

One loss, applied everywhere:

$$
\mathcal{L} = -\sum_t \log p_\theta(x_t \mid x_{<t})
$$

For every position `t` in every training sequence, predict the token there given everything before. Average the log-loss. Backpropagate. Update weights. Repeat for trillions of tokens.

That's it. Pretraining has one job: **next-token prediction on a lot of text.**

### 2.2 The data

Frontier models train on **multi-trillion-token corpora**:

- GPT-3 (2020): ~300B tokens.
- Llama 1 (2023): 1.4T.
- Llama 3 (2024): 15T.
- GPT-4, Claude, Gemini: 10T+ (not officially disclosed).

Sources (approximate mix for an open model):
- Web crawl (Common Crawl, FineWeb): 60–80%. Filtered heavily for quality.
- Code (GitHub, StackOverflow): 5–15%. Critical for reasoning even on non-code tasks.
- Books: 5–15%.
- Academic papers (arXiv, PubMed): 1–5%.
- Wikipedia: 1–3%.
- Synthetic data (generated by other models): growing — 20%+ in recent models.

Data quality matters more than quantity past a point. A 1T-token curated dataset outperforms a 5T-token uncurated one. Curation involves:
- Deduplication (exact and near-duplicate).
- Filtering boilerplate, low-quality web junk.
- Language identification (multilingual balance).
- Toxicity filtering (lightly; aggressive filtering hurts capability).
- Quality scoring (fasttext classifiers, perplexity filters).

### 2.3 Compute scale

Pretraining a 70B model on 15T tokens:
- ~6 × 15T × 70B = **6.3 × 10²⁴ FLOPs** (the "Chinchilla" formula).
- On 10k H100s at ~2 exaFLOPs aggregate FP8: ~40 days.
- At $2/GPU-hour: ~$20M in compute alone. Add engineering, data, failures, reruns: $50–100M for a frontier pretrain.

The compute scale is why only a handful of labs train frontier models from scratch.

### 2.4 What pretraining produces

A **base model**: excellent at continuing text, terrible at following instructions.

Prompt: `"Write a haiku about the ocean."`

Base model's likely completion: `"Write a haiku about the mountain. Write a haiku about the sky. Write a haiku about..."` — because in its training data, that sentence most often appeared as part of a list of prompts, not as an instruction to be fulfilled.

The base model has absorbed facts, grammar, reasoning patterns, even code. It just doesn't know how to behave like a helpful assistant. That's what comes next.

### 2.5 Scaling laws (the Chinchilla moment)

Pre-2022 instinct: bigger models → better models. Train a 500B-parameter model on 300B tokens and you'll beat a 200B model.

DeepMind's "Chinchilla" paper (2022) said: no. For a given compute budget, the optimal model is *smaller* and trained on *more tokens*. The ratio: roughly 20 tokens per parameter.

- 70B model → train on ~1.4T tokens for Chinchilla-optimal.
- 175B GPT-3 (trained on 300B) → under-trained. Would be beaten by a 60B model on 1.2T tokens.

Post-Chinchilla, labs train smaller models on more data. Llama 3 8B is trained on **15T tokens** — 1875 tokens per parameter, deliberately over-trained beyond Chinchilla for inference efficiency. When you serve a model, inference cost scales with parameter count, not training cost. A smaller, over-trained model is cheaper to serve and often better.

## 3. Stage 2: Supervised Fine-Tuning (SFT)

### 3.1 The goal

Teach the base model the **format** of being an assistant:

- Turn-based dialogue (`user:` / `assistant:`).
- Answer the question that was asked.
- Refuse harmful requests.
- Use tools when presented.
- Format code in code blocks.
- Don't go on forever.

### 3.2 The data

Curated (instruction, response) pairs. Sizes from 10k to 1M examples. Quality >>> quantity. Sources:

- **Human-written** (expensive, high quality). Teams of subject-matter experts write ideal responses.
- **Synthetic** (from stronger models). Generate responses with GPT-4 or Claude, filter for quality. Most open models heavily use synthetic SFT data.
- **Distilled** from the same model at a previous stage (self-improvement).
- **Public datasets** (Alpaca, Dolly, OpenAssistant, UltraChat).

Example SFT record:
```json
{
  "messages": [
    {"role": "system", "content": "You are a helpful, harmless, honest assistant."},
    {"role": "user", "content": "Explain recursion in one paragraph."},
    {"role": "assistant", "content": "Recursion is a technique where a function calls itself..."}
  ]
}
```

### 3.3 The training objective

Same as pretraining — predict the next token — but loss is computed **only on the assistant's tokens**. The user turn's tokens are in the context but not scored. This teaches the model to produce the assistant portion, not to regenerate the question.

```python
# Conceptually:
for message in messages:
    if message.role == "assistant":
        loss += next_token_loss(message.tokens)
    # else: just feed them in as context, no loss
```

### 3.4 What SFT teaches, specifically

- **Turn structure.** The chat template becomes natural.
- **Instruction following.** "Given a task, do the task, don't just continue the text."
- **Refusal.** When asked for harmful content, respond with a refusal (trained on explicit refusal examples).
- **Format compliance.** JSON when asked, tables when asked, citations when asked.
- **Tool use.** Emitting the specific token sequences that trigger tool calls.
- **Calibration of uncertainty.** Saying "I don't know" when appropriate (hard; imperfectly solved).

### 3.5 SFT limitations

SFT can teach the model what a good answer *looks like*, but it can only teach from the specific examples in the training set. It can't tell the model "this answer is better than that answer on dimensions you haven't seen before." For that, you need preferences.

---

## 4. Stage 3: Preference tuning

The model now sounds like an assistant. But on many axes — helpfulness, conciseness, honesty, tone — SFT doesn't fully align it. This is where RLHF, DPO, and constitutional AI come in.

### 4.1 The key insight

It's much easier for a human to say **"A is better than B"** than to write an ideal answer from scratch. Preferences are cheaper than demonstrations, and they can be applied to *any* pair of outputs the model produces.

---

## 5. RLHF — Reinforcement Learning from Human Feedback

### 5.1 The classic recipe (InstructGPT, 2022)

Three steps:

**5.1.1 Collect preference data.** For each prompt, have the SFT model generate 2–4 candidate responses. A human ranks them best to worst.

```
Prompt: "Explain recursion."
A: [detailed, good example]       ← chosen
B: [shallow, confusing]           ← rejected
```

Collect tens to hundreds of thousands of such pairs.

**5.1.2 Train a reward model.**

A separate model (initialized from the SFT model, modified to output a scalar reward) is trained on the preference pairs:

$$
\mathcal{L}_{RM} = -\log \sigma(r_\theta(x, y_w) - r_\theta(x, y_l))
$$

"Make the chosen response's reward higher than the rejected response's reward." The reward model learns to predict what a human would rate highly.

**5.1.3 RL fine-tune the policy against the reward.**

Use PPO (Proximal Policy Optimization) or a variant. The policy model generates responses; each response is scored by the reward model; PPO updates the policy to increase expected reward.

With a **KL penalty**: don't let the model drift too far from the original SFT model. Without this constraint, the policy finds reward-hacking exploits — generating gibberish that happens to score high on the reward model.

$$
\mathcal{L}_{RL} = \mathbb{E}[r_\phi(x, y)] - \beta \cdot D_{KL}(\pi_\theta \| \pi_{SFT})
$$

### 5.2 Why RLHF is hard

- **Reward hacking.** The reward model is a proxy for human preference; the policy will exploit errors in the proxy. Needs careful reward model training and constant evaluation.
- **Instability.** PPO is finicky. A hyperparameter off by 2× can collapse the model to degenerate outputs.
- **Cost.** Human preference labeling is expensive. Reward model training is its own infra. PPO requires storing the policy, reference model, reward model, and sometimes a critic in memory simultaneously — 4× the memory of SFT.

RLHF was the breakthrough that turned GPT-3 into ChatGPT. It also broke plenty of models during development.

## 6. DPO — Direct Preference Optimization

In 2023 (Rafailov et al.), a cleaner alternative emerged: **skip the reward model and PPO entirely.**

### 6.1 The insight

You can derive a closed-form loss that optimizes the same objective as RLHF, directly on preference pairs, without an intermediate reward model:

$$
\mathcal{L}_{DPO} = -\log \sigma\left(\beta \log \frac{\pi_\theta(y_w \mid x)}{\pi_{ref}(y_w \mid x)} - \beta \log \frac{\pi_\theta(y_l \mid x)}{\pi_{ref}(y_l \mid x)}\right)
$$

In English: "increase the log-probability ratio of the chosen response (vs. the reference model) more than the rejected response's ratio."

### 6.2 Why DPO wins in practice

- **Simpler.** One training loop, no reward model.
- **More stable.** No reward hacking loop.
- **Cheaper.** ~3× less compute than PPO-RLHF.
- **Competitive quality.** On most benchmarks, DPO matches or nearly matches RLHF.

Most open models released after 2023 (Zephyr, Tulu, many Llama fine-tunes) use DPO. Some frontier labs still use variants of PPO because they've invested in the infrastructure.

### 6.3 Variants of DPO

- **IPO** (Identity Preference Optimization): changes the objective slightly to reduce overconfidence.
- **KTO** (Kahneman-Tversky Optimization): works with *unpaired* feedback — "this response is good/bad" instead of "A > B".
- **ORPO**: combines SFT and preference tuning in one stage.
- **SimPO**: simplifies DPO further, drops the reference model.

These are incremental refinements. DPO is the family; the variants tune details.

## 7. Constitutional AI and RLAIF

### 7.1 Scaling preferences with AI

Human labeling doesn't scale to billions of comparisons. Anthropic introduced **Constitutional AI (CAI)** — use an LLM as the preference labeler.

Recipe:
1. Define a "constitution" — a list of principles (be helpful, avoid harm, be honest, etc.).
2. Generate responses to prompts.
3. For each response, have an LLM critique it against the constitution and revise.
4. Build preference data from (original, revised) pairs.
5. Train with RLHF or DPO on this AI-generated preference data.

**RLAIF (Reinforcement Learning from AI Feedback)** is the general term for this: reward model trained on AI-labeled preferences.

### 7.2 Why it works

- Scales cheaply to millions of comparisons.
- The critic LLM can be more systematic than tired human labelers.
- Multiple critics (ensemble) reduce bias.

Downsides:
- Inherits the critic's biases.
- Can amplify sycophancy if the critic likes flattering responses.
- Requires careful constitution design and auditing.

### 7.3 Claude's approach

Anthropic's Claude series uses Constitutional AI as a core part of training. The published constitution is a short document of principles; the model is trained to apply them to its own outputs. The "helpful, harmless, honest" framing came from this.

### 7.4 RLHF vs RLAIF in practice

Modern frontier training is **hybrid**:
- High-stakes, high-ambiguity tasks: human preferences (safety, harmful content, edge cases).
- High-volume, well-specified tasks: AI preferences (clarity, format, standard reasoning).

This is cheaper and faster than pure RLHF while keeping humans in the loop on the things that matter most.

## 8. Reasoning-specific training (the 2024–2026 wave)

"Reasoning models" (o1, o3, Claude extended thinking, DeepSeek-R1) have a specialized post-training stage beyond preference tuning.

### 8.1 The goal

Teach the model to generate long internal "thinking" chains before its final answer — solving problems step-by-step, sometimes for thousands of tokens, and improving as it goes.

### 8.2 The training signal

Instead of preference over final answers, reward is based on whether the **final answer is correct** on verifiable tasks:

- Math problems (correct numeric answer).
- Code (passes unit tests).
- Logic puzzles (provable solution).

The model generates a long chain of reasoning; if the final answer is correct, positive reward; if wrong, negative. Critically, the *intermediate* tokens are not judged — the model is free to explore, backtrack, try multiple approaches.

This is closer to classic RL than RLHF — the reward signal is **ground truth**, not human preference.

### 8.3 The DeepSeek-R1 recipe (2025, fully published)

1. Start with a strong base model.
2. **Pure RL** on math/code tasks with ground-truth rewards — no SFT. Model learns to reason by trial and error.
3. Observe emergent "aha" moments: the model learns to pause, re-check, try alternatives. Nobody taught this; it emerged from reward pressure.
4. Mix in SFT on human-written reasoning chains to smooth out formatting.
5. More RL, and distill back into smaller models.

This recipe moved the needle on reasoning benchmarks by double-digit points. Every major lab has adopted a version of it.

### 8.4 What reasoning training adds

- Much better at multi-step problems (math, code, logic).
- Willingness to explore and backtrack.
- Longer, more deliberate outputs (and much higher token cost).
- Often *worse* at creative or open-ended tasks — the RL pressure biases toward "there is one right answer" thinking.

This is why reasoning models are offered **in addition to**, not instead of, regular models. OpenAI's o-series + GPT family; Anthropic's extended thinking mode toggled on top of Claude. You pay more for reasoning; you use it when you need it.

## 9. Distillation — how small models get good

Once a frontier model exists, you can train smaller models to mimic it.

### 9.1 Hard-label distillation

Use the big model to generate answers; train a small model via SFT on those answers. Simple. Common.

### 9.2 Soft-label distillation

Use the big model's **logits** (probability distribution over next tokens) as the training target for the small model:

$$
\mathcal{L}_{distill} = D_{KL}(p_{teacher}(\cdot \mid x) \| p_{student}(\cdot \mid x))
$$

Carries more information than hard labels — the small model learns the teacher's uncertainty, not just its top pick.

### 9.3 Why it works so well

Distillation is effective because modern big models are massively over-parameterized for any given task. A small student can absorb most of the capability on specific tasks while being 10–100× cheaper.

Examples:
- **Phi-3** (Microsoft, 2024): 3.8B params, trained on curated + synthetic data distilled from GPT-4. Matches Llama 3 8B on many benchmarks.
- **Gemma 2** (Google): distilled from larger Gemini variants.
- **Mistral Small, OpenAI GPT-4o mini, Claude Haiku**: smaller members of their families, partially distilled.

Limitation: distillation preserves the teacher's behavior, including mistakes. Students can't exceed the teacher on the distilled distribution.

## 10. Alignment is not "done"

After all three stages, models still exhibit:

- **Sycophancy.** Agreeing with the user even when wrong. Stems from RLHF's "users prefer agreeing responses."
- **Refusal calibration errors.** Refusing safe requests, accepting unsafe ones with clever framings. Hard to tune perfectly.
- **Hallucination.** Confidently producing false information. Pretraining does not instill epistemic humility.
- **Format brittleness.** Small prompt changes produce large behavior changes. Deep fix is architectural, not post-training.
- **Reward gaming.** The model learns to satisfy the proxy, not the intent. A reward model that rewards "thorough" answers → model produces needlessly long answers.

Every new model release ships with preference-tuning fixes for the most visible of these. None of them are fully solved.

---

## 11. Continual / online training

Most frontier models are **frozen** at release — the weights don't update from user interactions. This is deliberate:

- Prevents prompt injection poisoning the model.
- Keeps behavior consistent and auditable.
- Simpler to version and deploy.

"Continual learning" during serving is research-grade. What production does instead:

- **Periodic re-training** with new data, every 6–12 months.
- **Context-level adaptation** (RAG, memory, in-context examples) — the model stays the same; its context changes.
- **Fine-tunes on demand** for customers with enough data and need (OpenAI fine-tuning, Anthropic custom fine-tuning when available).

## 12. What each stage *teaches*, summarized

| Stage | Primary gain | Typical duration | Data size |
|---|---|---|---|
| Pretraining | World knowledge, grammar, coding syntax, basic reasoning | Weeks to months | Trillions of tokens |
| SFT | Instruction following, chat format, tool use format | Hours to days | 10k–1M examples |
| Preference tuning (RLHF / DPO / RLAIF) | Helpfulness, conciseness, tone, refusal calibration | Hours to days | 100k–10M preference pairs |
| Reasoning RL (optional) | Multi-step reasoning, self-correction | Days | Problems with verifiable answers |
| Distillation (separate, for small models) | Capability compression | Days | Teacher-generated responses |

When a model behaves poorly, you can often guess which stage is responsible:

- Gets facts wrong → pretraining coverage or hallucination tuning.
- Ignores formatting instructions → SFT gap.
- Refuses safe things → preference tuning miscalibration.
- Can't multi-step reason → no reasoning RL, or too small a base.
- Shallow on technical topics → distilled from a teacher that was also shallow there.

---

## 13. Summary

Training a frontier LLM has three main stages:

1. **Pretraining**: next-token prediction on trillions of tokens. Slow, massively expensive, produces a completion machine.
2. **SFT**: supervised fine-tuning on curated examples. Teaches the model how to behave as an assistant.
3. **Preference tuning** (RLHF / DPO / RLAIF / constitutional AI): aligns the model with human-preferred outputs.

Add **reasoning RL** for models with extended thinking. Add **distillation** to make cheap small models from expensive big ones.

Each stage is a continuation of the same neural network — new data, different loss, fewer examples at each step. The base model is the reservoir of knowledge; SFT sculpts it; preference tuning polishes it.
